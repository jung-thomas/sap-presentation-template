# v0.4.4 — VR test infrastructure repair

**Status:** Draft for review
**Date:** 2026-06-18
**Author:** Thomas Jung (with Claude assistance)
**Supersedes:** none — closes the v0.4 series before v0.5
**Scope decision:** Option 1 (minimal-fix release) — see §3.

## 1. Summary

v0.4.4 is a **test-infrastructure repair release**. No new layouts, no brand changes. It fixes a single root-cause bug that has silently broken the visual-regression test suite for the entire v0.4 cycle, expands VR coverage to the full kitchen-sink deck, and rebaselines all snapshots against the (now correctly-themed) gallery.

The fix is a **one-character change** to the gallery deck's front-matter; everything else in this spec is consequence (rebaseline, coverage bump, deletion of the now-redundant bi/o investigation).

## 2. Background — what the audit found

While investigating the inherited `bi/o` icon-resolution flake (originally scoped as one of three v0.4.4 items), Chrome DevTools network inspection of the gallery dev server showed every slide importing `@slidev/client/layouts/default.vue` instead of the project's `theme/layouts/*.vue`.

Tracing through Slidev's plugin source:

- Slidev's components plugin scans `dirs: [clientRoot/builtin, ...roots.map(i => i + '/components')]`. The "roots" array includes the active theme root.
- Slidev resolves `theme: <path>` from front-matter **relative to the markdown file's directory**, not the project root.
- `pages/all-layouts.md` declares `theme: ./theme`. Slidev resolves this as `pages/theme/` — which doesn't exist — and silently falls back to the built-in default theme.
- `slides.md` (project root) declares `theme: ./theme` correctly resolving to `theme/`, so it works.

**Consequences:**
1. Every gallery slide rendered with `default.vue` — none of the SAP layouts (cover, divider, thank-you, agenda, text-with-icons, table, q-and-a, …) loaded.
2. Every committed `tests/visual.spec.ts-snapshots/slide-NN.png` shows blank-white content, raw front-matter prose, or only the slide-overview drawer — not actual themed slide rendering. `maxDiffPixelRatio: 0.05` made "blank vs. blank" pass trivially. **The VR test suite has been a silent no-op since the gallery deck was created.**
3. The `<Bio>` tag flake was a side-effect: when the theme isn't loaded, `theme/components/Bio.vue` isn't scanned, so unplugin-vue-components reports the tag unresolved. unplugin-icons (configured by Slidev with empty `prefix: ""`) then greedily reinterprets `<Bio>` as `<bi:o>` — Bootstrap Icons collection `bi`, icon `o` — which doesn't exist. **Fixing the theme path resolves the bi/o flake as a side-effect, with no separate registration work needed.** Verified empirically.

The fix is `theme: ../theme` in `pages/all-layouts.md`.

## 3. Scope decision: Option 1 (minimal-fix release)

Three options were considered:

| | Scope | Outcome |
|---|---|---|
| **Option 1** | Theme path + rebaseline + 101→146 coverage. Fix bi/o flake as a side-effect of theme path. | Closes both regressions. Baselines real for the first time. Audit-driven layout repairs become a follow-up release. |
| Option 2 | Option 1 + fix every drift surfaced by the new baselines | Comprehensive but unbounded scope; what the new baselines reveal is unknown. |
| Option 3 | Hotfix-only release (v0.4.3.1) for theme path + bi/o; defer coverage + audit to v0.4.4 | Two releases for one logical fix; gallery slides 102–146 stay unprotected longer. |

**Selected:** Option 1.

Rationale: the theme path bug is severe enough that the release should be primarily *about* fixing it. Audit-driven layout repairs become a v0.4.5 (or v0.5) spec once the rebaseline reveals what's actually drifting against POTX. Two releases is overhead.

## 4. Goals

1. **Restore VR test fidelity** — gallery deck renders with project theme; baselines reflect the actual slide content.
2. **Close the coverage gap** — VR test sweeps all 146 gallery slides, not the first 101.
3. **Eliminate the bi/o flake** — verified that fixing the theme path resolves it; no separate work item required.
4. **Document the trap** — leave a comment in `pages/all-layouts.md` explaining why `../theme` (not `./theme`), so future authors don't "fix" it back.
5. **No regression in `slides.md` rendering** — the main deck must continue to work exactly as before.
6. **Produce a findings file** — `docs/superpowers/findings/2026-06-18-v0.4.4-rebaseline-findings.md` is a deliverable of this release. Even if the spot-check surfaces nothing, an explicit "no findings" file is a useful artifact for the v0.4.5 / v0.5 spec to reference. See §8.1.
7. **Opportunistic doc hygiene** — `tests/README.md` currently says "0.5% pixel-diff tolerance" and "0.5% `maxDiffPixelRatio`" in two places, but `playwright.config.ts` has been at `0.05` (5%) since the v0.3 rebaseline. The plan corrects both numbers in the same touch.

## 5. Non-goals

- **Visual fidelity repairs against POTX.** The new baselines may reveal layout drift that POTX-fidelity locked baselines would have caught. Filing those as findings is in scope; fixing them is **explicitly deferred** to v0.4.5 / v0.5.
- **Investigating why the gallery deck was created with the wrong path.** Git archaeology won't change the fix. (The likely explanation: when `pages/all-layouts.md` was scaffolded, `./theme` was copy-pasted from `slides.md`'s front-matter without adjusting for the relative-path semantics.)
- **Restructuring the gallery deck or the theme.** Both stay where they are.
- **Tightening `maxDiffPixelRatio`.** Stays at `0.05`. Tightening it would surface font-hinting noise from the cross-platform baseline situation (Windows local vs. Linux CI) and is its own scope.
- **Investigating other Slidev path-resolution gotchas.** Out of scope.

## 6. The fix

### 6.1 Front-matter change

In `pages/all-layouts.md`:

```diff
 ---
-theme: ./theme
+theme: ../theme
 title: 'Kitchen-Sink Gallery — All Layouts & Components'
 info: |
   Visual regression reference deck.
```

Add an inline comment so the gotcha is documented in-place:

```yaml
---
# Slidev resolves `theme:` relative to *this markdown file's directory*, not
# the project root. From pages/, the theme lives one level up. Don't "fix"
# this back to `./theme` — that resolves to pages/theme/ (which doesn't
# exist) and silently falls back to Slidev's built-in default theme,
# breaking every layout in this gallery and all VR baselines built from it.
# Verified by inspecting the compiled output of pages/all-layouts.md__slidev_*.md.
theme: ../theme
title: 'Kitchen-Sink Gallery — All Layouts & Components'
...
```

(YAML comments are valid in Slidev's front-matter parser; verified in dev.)

### 6.3 `theme/setup/_dataSources.ts` absolute-path fix (added during implementation)

**Discovered during Task 4 (rebaseline) execution.** Once the theme path fix in §6.1 lands, the gallery starts loading the theme — and the theme imports `theme/setup/_dataSources.ts`, which uses absolute-from-Vite-root imports for the YAML data files at the project root:

```ts
// theme/setup/_dataSources.ts (before)
const presenterModules = import.meta.glob('/presenters/*.yaml', { eager: true, import: 'default' })
const teamModules     = import.meta.glob('/teams/*.yaml',      { eager: true, import: 'default' })
const programModules  = import.meta.glob('/programs/*.yaml',   { eager: true, import: 'default' })
import eventData from '/event.yaml'
```

Slidev sets Vite's user-root to `dirname(entry)` (verified in `node_modules/@slidev/cli/dist/chunk-HOVIRHCR.js`). For `slides.md` (project root) this resolves `/` to the project directory, and the imports work. For `pages/all-layouts.md` (one level deeper) Vite resolves `/` to `pages/` — so `/event.yaml` becomes `pages/event.yaml` (doesn't exist) and the same for the globs. Vite throws an import-analysis error overlay on every gallery slide, blocking the rendered slide content.

This bug existed before v0.4.4 but was invisible because the gallery wasn't loading the theme. The §6.1 fix exposed it.

**Fix:** change all four imports in `_dataSources.ts` to **paths relative to the importing file**, anchored at `theme/setup/`:

```ts
// theme/setup/_dataSources.ts (after)
const presenterModules = import.meta.glob('../../presenters/*.yaml', { eager: true, import: 'default' })
const teamModules     = import.meta.glob('../../teams/*.yaml',      { eager: true, import: 'default' })
const programModules  = import.meta.glob('../../programs/*.yaml',   { eager: true, import: 'default' })
import eventData from '../../event.yaml'
```

Relative paths in `import.meta.glob` and ES `import` statements resolve against the importing file, not Vite's user-root, so the fix works for both decks regardless of where the entry markdown lives.

**Verification:** after the fix, the gallery dev server boots without the import-analysis error overlay; gallery slides render with actual themed content. Existing `slides.md` rendering is unchanged because the relative path resolves to the same target. The unit tests in `theme/setup/data.test.ts` mock `./_dataSources` (per the existing comment in the file), so they're unaffected.

### 6.4 `theme/vite.config.ts` + `SETTLE_MS` bump (added during implementation, second amendment)

**Discovered during the second attempt at Task 4 rebaseline.** With §6.1 + §6.3 in place, the gallery was finally importing `theme/setup/_dataSources.ts` with correct relative paths — but the YAML imports themselves still failed to parse. Vite logged `Failed to parse source for import analysis because the content contains invalid JS syntax. You may need to install appropriate plugins to handle the .yaml file format`, and every slide that depends on presenter / event / program / team data (cover, divider, thank-you, q-and-a, Bio component) showed Slidev's slide-error fallback.

Tracing through Slidev's source (`node_modules/@slidev/cli/dist/chunk-YP37OZJY.js: resolveViteConfigs`), Slidev only loads `vite.config.ts` from each path in its `roots` array, where `roots = uniq([...themeRoots, ...addonRoots, userRoot])`. For `pages/all-layouts.md`, userRoot is `pages/` — and `pages/vite.config.ts` doesn't exist. **The project-root `vite.config.ts` (which declares `@rollup/plugin-yaml`) is therefore never loaded for the gallery deck.**

This was invisible in v0.3 / earlier v0.4 because the gallery wasn't loading the theme at all (§6.1 bug). The §6.1 fix surfaced a §6.3 path bug; §6.3 fix surfaced this YAML-plugin bug. Each layer peeled back exposed the next.

**Fix #1 — `theme/vite.config.ts`:** add a Vite config at the theme root that registers `@rollup/plugin-yaml`. The theme dir is always in Slidev's `roots`, so this config is loaded regardless of which deck consumes the theme:

```ts
// theme/vite.config.ts (new)
import { defineConfig } from 'vite'
import yaml from '@rollup/plugin-yaml'

export default defineConfig({
  plugins: [yaml()]
})
```

The project-root `vite.config.ts` keeps its existing `yaml()` registration unchanged; Vite plugins are idempotent when the same plugin is registered twice (rollup-pluginutils' `createFilter` returns true/false the same way regardless), so for `slides.md` (where both configs apply) this is a no-op duplicate. For `pages/all-layouts.md`, only the theme config applies — that's the one that matters.

**Fix #2 — `SETTLE_MS` bump:** with §6.1 + §6.3 + §6.4 fix #1 all in place, the gallery now renders real themed content — but slides that load YAML data via `import.meta.glob` need a few hundred milliseconds for Vue to mount the components after the data resolves. The existing `tests/visual.spec.ts` settle of `200ms` was tuned against the previously-broken (= already-blank) rendering and is too short for real content. Empirically, 3000ms reliably catches the rendered Bio component on a Bio-bearing slide; 200ms reliably misses it.

Bump `SETTLE_MS` from `200` to `2000`. Conservative middle ground — well over what's needed in practice, well under what would slow CI noticeably (146 slides × +1.8s = ~4.4 minutes added wall-clock; the rebaseline was already 5–10 minutes).

**Verification:** after both fixes, the test scaffolding produces real themed screenshots: slide /1 shows the kitchen-sink cover with date/classification, slide /74 shows the styled four-column table with classification footer, slide /82 shows the Bio component with avatar/name/title/bio/social badges plus the SAP brand-color gradient line. Verified empirically with a temporary debug spec; the spec was deleted after verification.

## 8. Rebaseline

After §6 and §7 land:

```bash
npm run test:visual:update
```

This regenerates all `tests/visual.spec.ts-snapshots/slide-NN.png` (1–146) against the corrected, themed gallery. **All 101 existing baselines will change** — they previously captured the unthemed default layout; they'll now capture the SAP-themed layouts. **45 new baselines** (slides 102–146) will be created.

The rebaseline must be run on Windows (current development OS) to match the existing snapshot platform convention documented in `tests/README.md §"Cross-platform baselines"`. The CI Linux job will likely flag font-hinting noise on the first run after this change; the existing 5% diff tolerance should absorb it (per the same README guidance for prior rebaselines).

## Appendix A — investigation transcript

Brief log of the audit that uncovered the theme path bug, for future reference:

1. Reproduced the bi/o flake in Chrome DevTools — Vite error overlay on gallery boot.
2. Network log showed the failing request was `GET /@id/~icons/bi/o` from referer `all-layouts.md__slidev_82.md`.
3. `curl http://localhost:3031/all-layouts.md__slidev_82.md?import` showed compiled slide importing `__unplugin_components_0 from "/@id/~icons/bi/o"` — confirming icon-hijack of the `<Bio>` tag.
4. Initial hypothesis: `<Bio>` not auto-resolved because Slidev only scans `./components/`, not `theme/components/`. Recommended fix: explicit `app.component()` registration.
5. While investigating registration mechanics, noticed the same compiled file imported `InjectedLayout` from `@slidev/client/layouts/default.vue` — Slidev's *built-in* default, not `theme/layouts/text-with-icons.vue` or any project layout.
6. Cross-checked `slides.md` via `npm run dev` (port 3030): compiled slides correctly imported `theme/layouts/*.vue`. Different deck, different behavior — clear signal that the issue was deck-local.
7. Inspected Slidev source (`node_modules/@slidev/cli/dist/chunk-YP37OZJY.js`): components plugin scans `roots.map(i => i + '/components')`. `roots` includes the theme root, which depends on theme resolution. Theme path is resolved relative to the markdown file.
8. Confirmed: `pages/all-layouts.md` declares `theme: ./theme` → resolves to `pages/theme/` → doesn't exist → silent default fallback.
9. Tested fix: changed to `theme: ../theme`, restarted gallery, re-fetched compiled slide 82 → `import __unplugin_components_0 from "<worktree>/theme/components/Bio.vue"` ✅, `import InjectedLayout from "<worktree>/theme/layouts/...vue"` ✅. Vite error overlay gone.
10. Re-checked all VR baselines (`tests/visual.spec.ts-snapshots/slide-*.png`): every one I sampled (slides 1, 3, 27, 50, 73, 74, 80) showed blank-white or raw-front-matter / overview-drawer content, never themed slide rendering. Confirmed VR suite is a no-op.
11. Reverted the fix (this is the spec phase; implementation comes later). Worktree clean.
