# v0.4.3 — Text-with-icons + Table layouts

**Status:** Draft for review
**Date:** 2026-06-16
**Author:** Thomas Jung (with Claude assistance)
**Supersedes:** none — adds new layouts to v0.4.2.3
**POTX source:** `SAP_Corp.potx` → `ppt/slides/slide9.xml` (Text with icons), `ppt/slides/slide11.xml` (Table – four columns)

## 1. Summary

v0.4.3 ships **two new content layouts** extracted from the POTX:

1. **`text-with-icons`** — 3×2 grid of icon + title + description + optional "Learn more" link, mirroring POTX slide 9. Auto-fits to 3, 4, 5, or 6 items based on the content count.
2. **`table`** — 4-column structured table with branded header (SAP-blue-7 fill, white text) and light-blue (sap-blue-4) cell borders, mirroring POTX slide 11. Cells accept Slidev-native markdown table syntax.

Both are defined entirely from POTX measurements (see appendix).

## 2. Goals

1. **Faithful POTX reproduction** — every measured value (positions, sizes, colors, fonts) within ±1% of POTX.
2. **Speaker-friendly authoring** — frontmatter-only API for `text-with-icons`; markdown body for `table`.
3. **Reuse existing primitives** — `<ClassificationFooter>`, `<SapIcon>` (existing component, 54-icon catalog at `theme/styles/_extracted/icons.json`), SAP brand tokens (`--sap-blue-*`, `--sap-font-family`). No new shared components.
4. **No new icon dependencies** — `<SapIcon>` is already wired to the extracted catalog from `scripts/extract-brand.mjs`. Authors reference icons by name string (e.g., `lightbulb`).

## 3. Non-goals

- Roadmap/timeline layout (slide 10) — deferred to v0.5+ (high-effort connector logic).
- Data points layout (slide 8), diagram-with-icons (slide 12), team-single (slide 13) — deferred.
- Custom POTX icon extraction — using the existing `<SapIcon>` catalog instead. If a brand-specific icon is missing later, extend the catalog via `scripts/extract-brand.mjs`.
- Internationalization — `en-US` only.

## 4. `text-with-icons` layout

### 4.1 POTX measurements (slide 9, 1920×1080)

| Element | Value |
| --- | --- |
| Title position | top-left, ~4% inset, ~24pt |
| Cell origin (col 1, row 1) | left: 4.1%, top: 22.2% |
| Cell width | 29.2% of slide |
| Cell content height | ~31% of slide (icon top y=22.2% → link bottom y=53.1%) |
| Column count | **3** (3 across × 2 down = 6 items) |
| Column gap | left edges at 4.1%, 35.4%, 66.6% — column gap ≈ 2% |
| Row gap | row 1 top 22.2%, row 2 top 60.6% — row gap ≈ 9% |
| Icon size | 4.1% × 7.3% (= 79×79 px on 1920×1080) |
| Icon→title gap | icon-bottom (29.5%) to title-top (30.9%) ≈ 1.4% whitespace |
| Title font | 16pt 72 Brand Medium, near-black |
| Description font | 14pt 72 Brand, black, 17.3%-tall block |
| Link "Learn more ›" | 14pt, color `#0070F2` (= sap-blue-7), 10.6% wide × 3.1% tall |

### 4.2 Architecture

Single layout file `theme/layouts/text-with-icons.vue`:

- Root `<div class="text-with-icons">`
- `<h1 class="text-with-icons-title">{{ fm.title }}</h1>` (optional — POTX has a small "Text with icons" eyebrow but real decks usually replace it with a section title; render only when `title:` set)
- `<div class="text-with-icons-grid" :class="`cols-${columnCount}`">`
  - `<div v-for="item in items" class="text-with-icons-cell">`
    - `<SapIcon :name="item.icon" class="text-with-icons-icon" />`
    - `<h3 class="text-with-icons-cell-title">{{ item.title }}</h3>`
    - `<p v-if="item.description" class="text-with-icons-cell-desc">{{ item.description }}</p>`
    - `<a v-if="item.link" :href="item.link.url" class="text-with-icons-cell-link">{{ item.link.text }} ›</a>`
- `<ClassificationFooter />`

Item-count → grid shape (locked rules — no per-implementation discretion):

| items | grid shape | rationale |
| --- | --- | --- |
| 1–3 | 1 row × N (N=items.length) | POTX cell width 29.2% accommodates up to 3 across with proper spacing |
| 4 | **2 rows × 2 cols** | 4-up across would shrink cells to ~22% (breaks icon/text proportions); 2×2 preserves POTX 29.2% cell width |
| 5 | 2 rows: top row 3 cells, bottom row 2 cells (centered) | preserves POTX cell width; bottom row left-padded to centered alignment |
| 6 | **3 cols × 2 rows** | POTX-faithful — slide 9 itself is exactly this shape |
| 7+ | 3 cols × ⌈items/3⌉ rows; cells overflow into row 3+ | preserves POTX proportions; vertical scroll/overflow not handled (deck author should split into multiple slides) |
| 0 | render title only + dev-mode console warning | safe default; no error |

### 4.3 Author API

```yaml
---
layout: text-with-icons
title: Why CAP?
items:
  - icon: lightbulb
    title: Productivity
    description: Less boilerplate, more declarative modeling.
    link:
      text: Learn more
      url: https://cap.cloud.sap
  - icon: people-connected
    title: Reuse
    description: Composable services and entities.
  # ...up to 6 items render POTX-faithfully (3×2); 7+ items overflow into row 3+
classification: PUBLIC
---
```

Required fields per item: `icon` (string, must exist in `theme/styles/_extracted/icons.json`) and `title` (string).
Optional per item: `description` (string), `link.text` + `link.url` (object), `iconColor` (CSS color overriding the default `var(--sap-blue-7)`).

Behavior with missing/empty `items`:

- `items` absent or `items: []` → render the title (if any) and a `<!-- text-with-icons: no items -->` HTML comment in dev mode for debugging. Production build: silently render empty grid, no error.

### 4.4 Icons

Use the existing **`<SapIcon>`** component at `theme/components/SapIcon.vue`, driven by the extracted catalog `theme/styles/_extracted/icons.json` (54 icons, including `lightbulb`, `people-connected`, `target-group` — the exact names this spec uses as examples).

API: `<SapIcon :name="lightbulb" size="79px" color="var(--sap-blue-7)" />`. Icon names must exist in the catalog. If a needed icon is missing, extend the catalog via `scripts/extract-brand.mjs` rather than adding new icon dependencies.

Icon color defaults to `currentColor`; the cell sets `color: var(--sap-blue-7)` so icons render in SAP brand blue. Authors can override per-cell via an `iconColor:` field on the item.

## 5. `table` layout

### 5.1 POTX measurements (slide 11, 1920×1080)

| Element | Value |
| --- | --- |
| Table position | left: 4.1%, top: 22.5% |
| Table size | 91.8% × 48.4% of slide |
| Column count | 4 (equal width 22.95% each = ~440 px) |
| Header row | height 8.1%, fill `#0070F2` (sap-blue-7), text `#FFFFFF`, 18pt 72 Brand Medium, center-aligned |
| Body rows | height 8.1% each (~87 px), text `#000000`, 14pt 72 Brand, center-aligned |
| Cell border | 1pt `#89D1FF` (sap-blue-4) on right + bottom edges |
| Cell padding | ~0.12" left/right (= ~14 px), ~0.06" top/bottom (= ~7 px) |

### 5.2 Architecture

Single layout file `theme/layouts/table.vue`:

- Root `<div class="layout table-layout">`
- `<h1 v-if="fm.title">{{ fm.title }}</h1>` (optional title — POTX has one but speakers can omit)
- `<div class="table-frame"><slot /></div>` — slot accepts the markdown body
- `<ClassificationFooter />`

The slot renders Slidev's standard markdown table → HTML `<table>`. Scoped CSS in `table.vue` styles `.table-frame > table` to match POTX:

```css
.table-frame :deep(table) {
  width: 100%;
  border-collapse: collapse;
  font-family: var(--sap-font-family);
}
.table-frame :deep(thead th) {
  background: var(--sap-blue-7);
  color: #ffffff;
  font-family: var(--sap-font-family-bold, var(--sap-font-family));
  font-weight: 600;
  font-size: 1.125rem;        /* 18pt ≈ 1.125rem */
  text-align: center;
  padding: 0.5rem 0.875rem;
  border: 1px solid var(--sap-blue-7);
}
.table-frame :deep(tbody td) {
  font-size: 0.875rem;         /* 14pt ≈ 0.875rem */
  text-align: center;
  padding: 0.5rem 0.875rem;
  border: 1px solid var(--sap-blue-4);
  color: var(--sap-text-primary, #000);
}
```

### 5.3 Author API

```yaml
---
layout: table
title: CAP vs. ABAP
classification: PUBLIC
---

| Capability | CAP | ABAP Cloud | RAP |
|------------|-----|------------|-----|
| Service modeling | CDS | CDS | CDS-based |
| Custom logic | Node.js / Java | ABAP | ABAP |
| Deployment | Cloud Foundry / Kyma | BTP ABAP env | S/4HANA |
| Best for | Greenfield apps | S/4 extensions | RAP-native apps |
```

The 4-column count is enforced by the **POTX visual style** — the layout doesn't reject 3- or 5-column markdown tables; they just won't have the POTX "exactly 4 equal columns" proportion. This is a soft constraint. Authors are expected to use 4 columns for the canonical POTX look.

## 7. Risks

| Risk | Impact | Mitigation |
| --- | --- | --- |
| Slidev's markdown plugin may not emit `<thead>`/`<tbody>` from a `\|---\|` divider, breaking the header-fill styling | Med | **Pre-impl gate (§6):** confirm `<thead>` emission against a stub slide before building styling. If absent, lock in either (a) styling `tr:first-child` instead of `thead th`, or (b) a frontmatter `headers: [...]` + `rows: [[...]]` schema. Decide BEFORE writing the layout, not at implementation time. |
| 7+ items overflow the slide vertically with no built-in clipping | Low | Document in spec (§4.2) that 7+ items should be split across slides; the layout doesn't enforce. Authors get a dev-mode warning if `items.length > 6` (informational, not error). |
| An author specifies an `icon` name that's not in `theme/styles/_extracted/icons.json` | Low | `<SapIcon>` already logs a console.warn on unknown names and renders no icon. Cell renders title + description without a leading icon — graceful degradation. |

## 8. Open questions

All deferred to implementation phase, not blocking spec approval:

1. **Slidev `<thead>` emission** — see §7 risk #1. Resolution: implementer runs a stub slide in `npm run dev` BEFORE building the table layout. If `<thead>` is emitted, use the spec's `:deep(thead th)` styling. If not, fall back to `:deep(tr:first-child th)`. Lock this decision in commit 1 of the table layout work.
2. **5-item grid centering** — the spec says "bottom row 2 cells centered." CSS implementation can use `grid-template-columns: repeat(3, 1fr)` plus `:nth-child(4)` and `:nth-child(5)` with `grid-column` placement (e.g., col 1 + col 2 of a 3-col grid, with the row left-padded to center the pair). The exact selector strategy is implementation detail.
