## 1. Summary

v0.2.0 shipped POTX-driven cover/divider/thank-you visual fidelity using **hand-rolled SVG approximations** of SAP brand decorations. After a brand-portal review (`brand.sap.com`) and a deeper strip-mine of `SAP_Corp.potx`, two structural problems surfaced:

1. **The POTX contains the official brand assets we approximated.** A 105 KB SVG named `id="Ripple_Pattern"` (the canonical SAP Anvil Ripple), seven recolored copies of a single **Flat Anvil primitive path**, the SAP wordmark/bookmark, three brand-licensed stock cover photos, an icon library, and the **embedded "72 Brand" typeface**. v0.2's parser ignored `ppt/media/` entirely.
2. **The POTX color/typography/layout systems are *named* and *self-curating*** — slide 20 embeds the Fiori Horizon palette as `(name, hex)` pairs (`Blue 7`, `#0070F2`) with text-color readability rules encoded by font color; slides 18/19/20 are HIDDEN tip slides; layout 45 is named `>DO NOT USE>`. v0.2 emitted anonymous tokens (`--brand-color-N`) and exposed deprecated layouts.

v0.3 rebuilds the brand layer from the POTX as ground truth. The work is one delivery in three architectural shifts:

- **Shift 1 — Asset reuse over re-creation.** Decorations compose the POTX's own SVG primitives instead of approximating new polygons. The Ripple Pattern, Flat Anvil, SAP wordmark, and brand-licensed stock photos ship verbatim.
- **Shift 2 — Named, semantic tokens.** Color tokens become `--sap-blue-7`, `--sap-mango-6`, `--sap-text-on-blue-7` etc., parsed from the POTX's own legend slide. Layouts named from specimen slide titles (`Roadmap`, `Quote`, `Meet our team`). Deprecated layouts (`>DO NOT USE>`, `>Copilot layouts >`) excluded from emission.
- **Shift 3 — Brand-compliance enforcement.** Logo clear-space rule (no decoration polygon enters a 1×logo-height keep-out box around the wordmark) is enforced by the layout component, not by the author's eye. Classification footer (`INTERNAL — SAP and External Parties under NDA Only`) is a first-class component with a hardcoded default matching the POTX.

Out of scope for v0.3: the official Dimensional Graphic Pattern (not in this POTX — would come from brand.sap.com later), 72 typeface served from a CDN (license-gated), embedded-font extraction from `.fntdata` (EOT format with root-binding bits), animation/transition fidelity, RTL.

## 2. Goals

1. **Decorations use POTX-sourced primitives.** Anvil Ripple (`image38.svg`) and the Flat Anvil polygon ship verbatim. No hand-rolled approximations.
2. **Color tokens are *named* and *paired*.** Authors write `color: var(--sap-blue-7)` and `background: var(--sap-blue-11)` and `color: var(--sap-text-on-blue-11)` — readability is a property of the token system, not author judgment.
3. **All 45 POTX layouts are accounted for.** Each layout either ships as a Vue layout, ships as a `variant:` of a Vue layout, or is *explicitly excluded* with a recorded reason (`>DO NOT USE>`, `>Copilot layouts >`).
4. **Logo clear-space is enforced.** No decoration is allowed to render geometry inside a `1×logo-height` keep-out box around the SAP shield. Slide 1 in v0.2.0 violated this; v0.3 makes it impossible.
5. **The classification footer is mandatory and overridable.** Default: POTX value. Override via deck-level `classification:` front-matter. Per-slide override allowed via `classification:` slide-level front-matter.
6. **Authoring API is additive.** Existing v0.2 covers (`variant: a..l`) keep working. New tokens, named variants, and components are additive — no breaking changes.

## 3. Non-goals

- **Dimensional Graphic Pattern** — not in this POTX, deferred until brand.sap.com asset is licensed for our use.
- **72 typeface from SAP CDN** — license-gated, would require a runtime config flag and an SAP-internal CDN endpoint. Inter remains the default fallback.
- **EOT font extraction** — embedded `.fntdata` files have root-binding bits and aren't legally servable as web fonts.
- **Templafy add-in fidelity** — `Spaker name - Dynamic` (sic) and similar Templafy field names are read into front-matter keys but we don't reproduce Templafy's runtime substitution beyond what front-matter already does.
- **think-cell, SmartArtPro chart fidelity** — `THINKCELLSHAPEDONOTDELETE` and `SP_POWERSHAPE` tags are explicitly *excluded* from the parser. Our charts use `@ui5/webcomponents-react-charts`.
- **PPTX import** — already shipped in v0.1; no v0.3 changes.
- **Animation/transition matching POTX** — out of scope as in v0.2.

## 4. Architecture

### 4.1 Five-layer model

| Layouts shipped | ~19 (one per POTX layout slot) | 45 mapped: ~25 Vue layouts × variants, 5 explicitly excluded |
| Decoration shapes | Hand-drawn SVG polygons | POTX-sourced (`image38.svg` Ripple, Flat Anvil primitive) |
| SAP wordmark | Brand Tools download | POTX `image37.png` (already aligned to layout coordinates) |
| Cover photos | Unsplash placeholder | 3 brand-licensed stock photos from POTX (`image40/41/86.png`) |
| Icons | None | 20+ POTX SVG glyphs available as `<SapIcon name="wrench" />` |
| Harvey Balls | None | `<HarveyBall :value="3" :of="8" />` (141 master shapes mined from POTX) |
| Classification footer | None | First-class `<ClassificationFooter />` with POTX default |

| Dynamic fields | None | `presenter`, `coPresenter`, `date`, `partnerLogo`, `event` front-matter mapped to POTX `*-Dynamic` shapes |
| Hidden slides | None | `hidden: true` front-matter (POTX uses `<p:sld show="0">` for tip slides) |

### 4.3 Decoration component model

A decoration is a single-responsibility Vue component receiving `(variant, image, classification)` props and rendering inline SVG. Decorations **never** position the SAP logo — that's the layout's job, and the layout enforces clear-space.

The decoration library v0.3 ships:

| Component | POTX source | Used by |
| --- | --- | --- |
| `RipplePattern.vue` | `<img src="/sap/anvil-ripple.svg" />` (image38) | Cover A, Divider C, Thank You B, Team band |
| `FlatAnvil.vue` | Inline SVG with canonical 605×297 path | Cover B, C, D, E, J, K, L (multi-shape compositions) |
| `AnvilGrid.vue` | Tiled instances of FlatAnvil | Team header band |
| `PhotoFrame.vue` | `<img>` with `image:` front-matter | Cover B, C, D, E (image takes a frame slot) |
| `SolidPanel.vue` | Single-color rectangle | Covers F, G, H, I (minimalist), Dividers A, B, D |
| `WordmarkBookmark.vue` | `<img src="/sap/wordmark.png" />` (image37) | Cover edge ornaments, Thank You corner |
| `GradientFade.vue` | CSS `linear-gradient` between two named tints | Cover L |

Decorations compose. Cover D = `PhotoFrame` + `FlatAnvil×3` + `WordmarkBookmark` (per the POTX specimen).

### 4.4 Color token emission

The token file `brand-tokens.css` is generated by parsing `ppt/slides/slide20.xml` (the POTX's own color legend). Each `(name, hex, font-color)` triple becomes three CSS variables:

```css
/* From slide20.xml: ('Blue 7', '#0070F2', white-text-on-it) */
--sap-blue-7: #0070F2;
--sap-text-on-blue-7: #ffffff;
--sap-blue-7-pair: #ffffff; /* alias */
```

Plus theme-scheme aliases parsed from `ppt/theme/theme1.xml`:

```css
--sap-accent-1: var(--sap-mango-6);   /* accent1 = #E76500 */
--sap-accent-2: var(--sap-teal-6);
--sap-accent-3: var(--sap-green-6);
--sap-accent-4: var(--sap-raspberry-6);
--sap-accent-5: var(--sap-pink-6);
--sap-accent-6: var(--sap-indigo-6);
--sap-link: var(--sap-blue-7);        /* hlink and folHlink both #0070F2 */
--sap-text-primary: var(--sap-grey-11);
--sap-text-secondary: var(--sap-grey-7);
```

The legacy v0.2 anonymous `--brand-color-N` tokens remain emitted as deprecated aliases (`--brand-color-1: var(--sap-blue-6); /* deprecated, use --sap-blue-6 */`) until the next major.

### 4.5 Layout manifest (`layouts.json`)

Every one of the 45 POTX layouts is enumerated with one of three statuses:

```jsonc
{
  "slideLayout1.xml":  { "status": "ship",    "vueLayout": "cover",     "variant": "a" },
  "slideLayout2.xml":  { "status": "ship",    "vueLayout": "cover",     "variant": "b" },
  // … all 12 cover variants (A–L) …
  "slideLayout15.xml": { "status": "ship",    "vueLayout": "divider",   "variant": "a" },
  "slideLayout30.xml": { "status": "ship",    "vueLayout": "quote" },
  "slideLayout31.xml": { "status": "ship",    "vueLayout": "qa" },
  "slideLayout45.xml": { "status": "exclude", "reason": ">DO NOT USE> — POTX self-deprecated" },
  "slideLayout38.xml": { "status": "exclude", "reason": ">Copilot layouts > — defer to v0.4" }
}
```

The manifest is consumed by `theme/setup/cover-variants.ts` (alias resolution) and by Storybook to generate the layout gallery.

### 4.6 Logo clear-space enforcement

The SAP brand mandates a keep-out box of `1× logo-height` around the wordmark on all sides (preferred; `0.5×` is minimum). The layout component computes the keep-out box from the cover-tokens (logo top/left/width as percentages of slide), then passes it down as a `clearSpace` prop:

```vue
<RipplePattern :clear-space="logoKeepOut" />
```

The decoration `<svg>` has a `<clipPath>` that punches the keep-out rectangle out of any geometry that would intersect it. Implementation: an SVG `mask` element painted black inside the keep-out, white elsewhere; the decoration's main `<g>` references the mask. Visual regression tests assert no decoration pixels are within the keep-out box (Playwright + a pixel-color test on a known coordinate).

### 4.7 Classification footer

The POTX hardcodes `INTERNAL — SAP and External Parties under NDA Only` at footer position 12 of every layout. v0.3 reproduces this with a layered default:

1. Per-slide front-matter `classification:` (highest priority)
2. Deck-level `classification:` in `slides.md`
3. POTX-extracted default (`INTERNAL — SAP and External Parties under NDA Only`)
4. Off if explicitly set to `null`

A `ClassificationFooter.vue` component lives at the slide-level (rendered by every layout). The string is wrapped in a `<small>` styled to match the POTX (8pt, grey-7, footer-position-12 = bottom-left at the cover-tokens-derived offset).

Authors choosing to publish externally (`PUBLIC`, `EXTERNAL`) are reminded by a CONTENT-GUIDE.md note that the default is INTERNAL and they need to explicitly downgrade.

### 4.8 Dynamic fields

Specimen slides reuse shapes named `Spaker name - Dynamic`, `Date - Dynamic`, `Partner Logo Placeholder`. These map to deck/slide front-matter:

| POTX shape | Front-matter | Default |
| --- | --- | --- |
| `Spaker name - Dynamic` (1st) | `presenter` | (empty) |
| `Spaker name - Dynamic` (2nd) | `coPresenter` | (empty) |
| `Date - Dynamic` | `date` | today (auto-filled at build) |
| `Partner Logo Placeholder` | `partnerLogo` | (none — slot empty if unset) |
| Title placeholder | `title` | required |
| Picture placeholder | `image` | (variant-dependent default) |

The `Speaker.vue` component v0.2 already handles `presenter:`. v0.3 extends it to `coPresenter:`, `date:`, `partnerLogo:` — covered by tests.

### 4.9 Hidden slides

POTX slides 18/19/20 (`Tips & tricks` author guidance) use `<p:sld show="0">`. v0.3 supports a `hidden: true` slide-level front-matter with the same semantic — slide is rendered in dev mode (with a banner: "This slide is hidden in published builds") but skipped at build time. Implementation: a `vite` plugin that prunes `hidden:` slides from the build bundle.

### 4.10 Tag exclude list

148 custom tags in `ppt/tags/*.xml` collapse to 5 names. Only `SP_AGENDA` carries brand information. The rest are third-party Office plugins:

```js
// scripts/lib/parse-potx-tags.mjs
const EXCLUDE_TAG_NAMES = [
  'SP_POWERSHAPE',           // SmartArtPro plugin
  'SP_LIBRARY',              // SmartArtPro library
  'CONTAINEDIMAGEPATH',      // Templafy local cache paths
  'THINKCELLSHAPEDONOTDELETE' // think-cell chart plugin
];
const KEEP_TAG_NAMES = ['SP_AGENDA'];
```

The `SP_AGENDA` value `"TOC TOCShowsSubsections Dividers SectionNumber SlideNumber"` becomes input to the agenda-layout config: TOC enabled, subsections shown, dividers between sections, section numbers, slide numbers.

## 5. Detailed Component Specs

### 5.1 `<RipplePattern>` — the canonical Anvil Ripple

```vue
<RipplePattern
  :color="'var(--sap-blue-6)'"   <!-- default; overridable -->
  :clear-space="logoKeepOut"     <!-- {top,left,width,height} % -->
  :placement="'bottom'"          <!-- top | bottom | full | hero-band -->
/>
```

Renders `image38.svg` (the 105 KB POTX file) inline (so the `fill` overrides work via CSS variables) clipped to the requested placement and masked by the clear-space box. The `id="Ripple_Pattern"` SVG attribute is preserved. Single-color via `currentColor`.

### 5.2 `<FlatAnvil>` — the brand primitive

```vue
<FlatAnvil
  :color="'var(--sap-blue-7)'"
  :viewBox="[605, 297]"
  :rotate="0"
/>
```

Renders the canonical path `M0 0 0 297 307.459 297 605 0 0 0 Z` (605×297 viewBox). Brand-rule check: `:rotate` is *only* allowed at 0° — rotating the anvil violates SAP brand. The prop exists for future-proofing but throws a dev-mode warning if non-zero.

### 5.3 `<AnvilGrid>` — Pattern 2 (Anvil Grid Pattern)

Tiles `FlatAnvil` instances on a configurable grid (default 12-col × 4-row, matching POTX slide 14 specimen). Two-color: `:darkColor` (Blue 11) for the band background, `:lightColor` (Blue 7) for the anvils. Used as a header band on Team layout.

### 5.4 `<HarveyBall>` — the rating glyph

```vue
<HarveyBall :value="3" :of="8" :color="'var(--sap-blue-7)'" />
```

Rendered as a circle with a sector pie filled from 0° to `(value/of) × 360°`. Sourced from POTX slide-master shapes `Harvey 0/N` through `Harvey N/N` (where N ∈ {3,4,8}). Three sizes via `:size="'sm'|'md'|'lg'"` (16/24/32 px).

### 5.5 `<SapIcon>`

```vue
<SapIcon name="wrench" :color="'var(--sap-blue-7)'" />
```

Resolves to `theme/styles/_extracted/media/icons/{name}.svg` (POTX `image28.svg` = wrench, `image50.svg` = family-care, `image70.svg` = chart, etc.). Catalog generated by `parse-potx-media.mjs` based on each SVG's `<g id="...">` semantic name (the POTX preserves these — `id="wrench"`, `id="family-care"`, `id="full-stacked-column-chart"`).

### 5.6 `<ClassificationFooter>`

```vue
<ClassificationFooter :level="'INTERNAL'" :scope="'SAP and External Parties under NDA Only'" />
```

Renders a styled footer matching POTX: 8pt, grey-7, footer-position-12 (bottom-left, x=footer-x%, y=98%, derived from cover-tokens). Levels: `PUBLIC`, `EXTERNAL`, `INTERNAL` (default), `CONFIDENTIAL`. Shorthand: `:level="'PUBLIC'"` renders `"PUBLIC"` only (no scope). `:level="null"` hides the footer entirely.

### 5.7 `<Bio>` — refit to POTX slide-14 composition

v0.2 ships a `<Bio>` component. v0.3 redesigns it to match POTX slide 14 (`Meet our team`):

- 4 cards in a row (responsive: 2×2 on narrow viewports)
- Each card: photo on top (4:5 aspect), name in `var(--sap-text-primary)` bold, role in `var(--sap-text-secondary)` regular
- Optional QR code slot below role (POTX includes one — for SAP business-card vCard / talk abstract URL)
- Cards positioned to *break* the AnvilGrid header band (top half on dark band, bottom half on white)

Authoring stays YAML-driven (existing `team.yaml` schema extended with `qr:` field).

## 6. POTX Extraction Pipeline (v0.3)

`scripts/extract-brand.mjs` orchestrates four parsers and three emitters.

```
SAP_Corp.potx
  │
  ├── parse-potx-palette.mjs     → palette.json   (53 named colors + on-color)
  ├── parse-potx-layouts.mjs     → layouts.json   (45 layouts × status)
  ├── parse-potx-specimens.mjs   → specimens.json (slide → layout mapping, dynamic field shapes)
  ├── parse-potx-media.mjs       → media/         (Ripple SVG, Flat Anvil, wordmark, photos, icons)
  │
  ├── emit-brand-tokens.mjs      → brand-tokens.css     (named CSS variables)
  ├── emit-layout-manifest.mjs   → layouts.json (final, with v0.3 status decisions)
  └── emit-cover-tokens.mjs      ← UNCHANGED from v0.2 (geometry only)
  └── emit-typography-tokens.mjs ← UNCHANGED from v0.2 (font sizes only)
```

Each parser/emitter is unit-tested with `vitest`. The full pipeline is integration-tested against `SAP_Corp.potx` (gitignored), with `describe.skipIf(!HAS_POTX)` for CI.

## 8. Migration from v0.2

- v0.2 decks keep working unchanged. Existing `variant: a..l` covers continue to work.
- v0.2's `--brand-color-N` tokens remain (deprecated aliases pointing to named tokens).
- Unsplash demo photo replaced with POTX `image40.png` cover (brand-licensed). The Unsplash photo stays in `public/covers/` as an alternate, documented in CONTENT-GUIDE.

## 9. Risks

| Risk | Likelihood | Impact | Mitigation |
| --- | --- | --- | --- |
| POTX image rights aren't clear for redistribution | Med | High | Confirm with SAP brand team before merging; if blocked, ship references to brand.sap.com URLs and let authors download |
| 105 KB inlined Ripple bloats every cover slide | Low | Med | Reference as `<img>` from public/, not inline; only inline when decorations need recoloring |
| Named-token churn breaks v0.2 decks | Low | High | Keep `--brand-color-N` deprecated aliases for one full minor cycle |
| `parse-potx-palette.mjs` regex-fragile against future POTX revisions | Med | Low | Pin against current POTX hash; warn on parse failure (don't fail build) |
| Logo clear-space `<mask>` breaks across browsers | Low | Med | Test in Chrome/Firefox/Safari/Edge; fallback to layout-level positioning if mask fails |
| Classification footer hard-codes English | Med | Low | i18n is non-goal for v1; document as a known gap |
| `image38.svg` inlining hits Vue template parser size limits | Low | Med | Serve as static asset, reference via `<img src>` when not recoloring |

## 12. Open questions

1. **Does v0.3 redistribute POTX-extracted media in the public template repo?** If not, the build pipeline must instead reference URLs on `brand.sap.com` (which require auth — breaks public-friendly goal). Recommendation: redistribute under SAP brand-tools terms, attributed in `THIRD-PARTY-NOTICES.md`. **Needs confirmation before implementation.**
2. **Should we ship 72 typeface as a non-default opt-in (env flag) in v0.3, or defer entirely to v0.4?** Recommendation: defer to v0.4, but ship the `--sap-font-family` token plumbing now so adding 72 later is one change.
3. **Visual regression — re-baseline whole suite, or only affected layouts?** v0.3 changes all covers/dividers/thank-you/bio. Recommendation: full re-baseline as a single PR commit, with `npm run test:visual:update` gated by user explicit approval.
