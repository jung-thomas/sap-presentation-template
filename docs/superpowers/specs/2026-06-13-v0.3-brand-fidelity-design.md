## 1. Summary

v0.2.0 shipped POTX-driven cover/divider/thank-you visual fidelity using **hand-rolled SVG approximations** of SAP brand decorations. After a brand-portal review (`brand.sap.com`) and a deeper strip-mine of `SAP_Corp.potx`, two structural problems surfaced:

1. **The POTX contains the official brand assets we approximated.** A 105 KB SVG named `id="Ripple_Pattern"` (the canonical SAP Anvil Ripple), seven recolored copies of a single **Flat Anvil primitive path**, the SAP wordmark/bookmark, three brand-licensed stock cover photos, an icon library, and the **embedded "72 Brand" typeface**. v0.2's parser ignored `ppt/media/` entirely.
2. **The POTX color/typography/layout systems are *named* and *self-curating*** — slide 20 embeds the Fiori Horizon palette as `(name, hex)` pairs (`Blue 7`, `#0070F2`) with text-color readability rules encoded by font color; slides 18/19/20 are HIDDEN tip slides; layout 45 is named `>DO NOT USE>`. v0.2 emits a hand-curated mix of `--sap-brand-{descriptor}` semantic names (`--sap-brand-blue`, `--sap-brand-blue-darker`, `--sap-brand-orange`) plus `--sap-color-{hex}` fallbacks for unmapped colors, and exposes deprecated POTX layouts. The semantic names are *handpicked descriptors*, not the POTX's own legend names — v0.3 replaces the curation step by parsing slide 20 directly.

v0.3 rebuilds the brand layer from the POTX as ground truth. The work is one delivery in three architectural shifts:

- **Shift 1 — Asset reuse over re-creation.** Decorations compose the POTX's own SVG primitives instead of approximating new polygons. The Ripple Pattern, Flat Anvil, SAP wordmark, and brand-licensed stock photos ship verbatim.
- **Shift 2 — Named, semantic tokens.** Color tokens become `--sap-blue-7`, `--sap-mango-6`, `--sap-text-on-blue-7` etc., parsed from the POTX's own legend slide. Layouts named from specimen slide titles (`Roadmap`, `Quote`, `Meet our team`). Deprecated layouts (`>DO NOT USE>`, `>Copilot layouts >`) excluded from emission.
- **Shift 3 — Brand-compliance enforcement.** Logo clear-space rule (no decoration polygon enters a 1×logo-height keep-out box around the wordmark) is enforced by the layout component, not by the author's eye. Classification footer (`INTERNAL — SAP and External Parties under NDA Only`) is a first-class component with a hardcoded default matching the POTX.

Out of scope for v0.3: the official Dimensional Graphic Pattern (not in this POTX — would come from brand.sap.com later), 72 typeface served from a CDN (license-gated), embedded-font extraction from `.fntdata` (EOT format with root-binding bits), animation/transition fidelity, RTL.

## 2. Goals

1. **Decorations use POTX-sourced primitives.** Anvil Ripple (`image38.svg`) and the Flat Anvil polygon ship verbatim. No hand-rolled approximations.
2. **Color tokens are *named* and *paired*.** Authors write `color: var(--sap-blue-7)` and `background: var(--sap-blue-11)` and `color: var(--sap-text-on-blue-11)` — readability is a property of the token system, not author judgment.
3. **All 45 POTX layouts are accounted for.** Each layout either ships as a Vue layout, ships as a `variant:` of a Vue layout, or is *explicitly excluded* with a recorded reason (`>DO NOT USE>`, `>Copilot layouts >`).
4. **Logo clear-space is enforced.** No decoration is allowed to render geometry inside a `1×logo-height` keep-out box around the SAP shield. Slide 1 in v0.2.0 violated this; v0.3 makes it impossible.
5. **The classification footer is mandatory and overridable.** Default: POTX value. Override via deck-level `classification:` front-matter. Per-slide override allowed via `classification:` slide-level front-matter.
6. **Authoring API is additive.** Existing v0.2 covers (`variant: a..l`) keep working. New tokens, named variants, and components are additive — no breaking changes.

## 3. Non-goals

- **Dimensional Graphic Pattern** — not in this POTX, deferred until brand.sap.com asset is licensed for our use.
- **72 typeface from SAP CDN** — license-gated, would require a runtime config flag and an SAP-internal CDN endpoint. Inter remains the default fallback.
- **EOT font extraction** — embedded `.fntdata` files have root-binding bits and aren't legally servable as web fonts.
- **Templafy add-in fidelity** — `Spaker name - Dynamic` (sic) and similar Templafy field names are read into front-matter keys but we don't reproduce Templafy's runtime substitution beyond what front-matter already does.
- **think-cell, SmartArtPro chart fidelity** — `THINKCELLSHAPEDONOTDELETE` and `SP_POWERSHAPE` tags are explicitly *excluded* from the parser. Our charts use `@ui5/webcomponents-react-charts`.
- **PPTX import** — already shipped in v0.1; no v0.3 changes.
- **Animation/transition matching POTX** — out of scope as in v0.2.

### 4.4 Color token emission

The token file `brand-tokens.css` is generated by parsing `ppt/slides/slide20.xml` (the POTX's own color legend). Each `(name, hex, font-color)` triple becomes two CSS variables:

```css
/* From slide20.xml: ('Blue 7', '#0070F2', white-text-on-it) */
--sap-blue-7: #0070F2;
--sap-text-on-blue-7: #ffffff;
```

**Canonical naming:** `--sap-{family}-{tint}` is the value, `--sap-text-on-{family}-{tint}` is the paired text color. Both are first-class — neither is an alias. Authors compose them as a pair (`background: var(--sap-blue-11); color: var(--sap-text-on-blue-11);`).

**Theme-scheme aliases** parsed from `ppt/theme/theme1.xml`:

```css
--sap-accent-1: var(--sap-mango-6);   /* accent1 = #E76500 */
--sap-accent-2: var(--sap-teal-6);
--sap-accent-3: var(--sap-green-6);
--sap-accent-4: var(--sap-raspberry-6);
--sap-accent-5: var(--sap-pink-6);
--sap-accent-6: var(--sap-indigo-6);
--sap-link: var(--sap-blue-7);        /* hlink and folHlink both #0070F2 */
--sap-text-primary: var(--sap-grey-11);
--sap-text-secondary: var(--sap-grey-7);
```

### 4.6 Logo clear-space enforcement

The SAP brand mandates a keep-out box of `1× logo-height` around the wordmark on all sides (preferred; `0.5×` is minimum). The layout component computes the keep-out box from the cover-tokens (logo top/left/width as percentages of slide), then passes it down as a `clearSpace` prop:

```vue
<RipplePattern :clear-space="logoKeepOut" />
```

The decoration `<svg>` has a `<clipPath>` that punches the keep-out rectangle out of any geometry that would intersect it. Implementation: an SVG `mask` element painted black inside the keep-out, white elsewhere; the decoration's main `<g>` references the mask.

**Visual regression test method:** the assertion is *not* a single-pixel sample. The test composites the rendered slide and asserts that **every pixel inside the keep-out rectangle equals the slide background color** (or transparent), with a tolerance of ≤2 pixels for sub-pixel anti-aliasing on the keep-out edges. Implementation: Playwright captures the slide as PNG, the test reads pixels in the keep-out region (computed from the same cover-tokens the layout uses), and fails if any non-edge pixel diverges from the expected background by more than the configured ΔE threshold (5.0 in CIE76). One test per shipped cover and divider variant. The threshold and edge-tolerance live in `tests/visual/clear-space.config.ts`.

### 4.7 Classification footer

The POTX hardcodes `INTERNAL — SAP and External Parties under NDA Only` at footer position 12 of every layout. v0.3 reproduces this with a layered default:

1. Per-slide front-matter `classification:` (highest priority)
2. Deck-level `classification:` in `slides.md`
3. POTX-extracted default (`INTERNAL — SAP and External Parties under NDA Only`)
4. Off if explicitly set to `null`

A `ClassificationFooter.vue` component lives at the slide-level (rendered by every layout). The string is wrapped in a `<small>` styled to match the POTX (8pt, grey-7, footer-position-12 = bottom-left at the cover-tokens-derived offset).

Authors choosing to publish externally (`PUBLIC`, `EXTERNAL`) are reminded by a CONTENT-GUIDE.md note that the default is INTERNAL and they need to explicitly downgrade.

### 4.9 Hidden slides

POTX slides 18/19/20 (`Tips & tricks` author guidance) use `<p:sld show="0">`. v0.3 supports a `hidden: true` slide-level front-matter with the same semantic — slide is rendered in dev mode (with a banner: "This slide is hidden in published builds") but skipped at build time. Implementation: a `vite` plugin that prunes `hidden:` slides from the build bundle.

### 4.10 Tag exclude list

148 custom tags in `ppt/tags/*.xml` collapse to 5 names. Only `SP_AGENDA` carries brand information. The rest are third-party Office plugins:

```js
// scripts/lib/parse-potx-tags.mjs
const EXCLUDE_TAG_NAMES = [
  'SP_POWERSHAPE',           // SmartArtPro plugin
  'SP_LIBRARY',              // SmartArtPro library
  'CONTAINEDIMAGEPATH',      // Templafy local cache paths
  'THINKCELLSHAPEDONOTDELETE' // think-cell chart plugin
];
const KEEP_TAG_NAMES = ['SP_AGENDA'];
```

The `SP_AGENDA` value `"TOC TOCShowsSubsections Dividers SectionNumber SlideNumber"` is parsed into a structured config object (`{ toc: true, showSubsections: true, dividers: true, sectionNumbers: true, slideNumbers: true }`) and **emitted into `theme/styles/_extracted/agenda-defaults.json`** alongside `layouts.json`. The `theme/layouts/agenda.vue` layout imports this JSON via Vite's `?json` suffix and uses it as the default config. Authors override per-deck via `agenda:` front-matter in `slides.md`:

```yaml
---
layout: agenda
agenda:
  toc: true
  showSubsections: false   # override POTX default
  dividers: true
  sectionNumbers: true
  slideNumbers: true
---
```

## 5. Detailed Component Specs

### 5.1 `<RipplePattern>` — the canonical Anvil Ripple

```vue
<RipplePattern
  :color="'var(--sap-blue-6)'"   <!-- default; overridable -->
  :clear-space="logoKeepOut"     <!-- {top,left,width,height} % -->
  :placement="'bottom'"          <!-- top | bottom | full | hero-band -->
  :inline="false"                <!-- default false; see below -->
/>
```

Default behavior (`:inline="false"`): renders `<img src="/sap/anvil-ripple.svg" />` referencing the static asset shipped to `public/sap/anvil-ripple.svg` (105 KB, cached, served as a single HTTP response). The image is positioned and clipped via CSS. The `:color` prop is **ignored** in this mode — the asset's built-in fill (`#1B90FF`, Blue 6) is used.

When `:inline="true"`: the SVG is loaded at build time via Vite's `?raw` suffix (`import RippleRaw from '/sap/anvil-ripple.svg?raw'`) and rendered with `v-html` (bypasses Vue template parser size limits). The decoration's wrapper applies a CSS variable that overrides the SVG's `fill` attribute via attribute selectors. Used only when the consumer needs to recolor the Ripple (e.g., a divider in a non-default brand color).

The `id="Ripple_Pattern"` SVG attribute is preserved in both modes. Clear-space masking via SVG `<mask>` works in inline mode; in `<img>` mode the layout positions the image so it never enters the keep-out box (cheaper, but requires no per-pixel mask).

### 5.2 `<FlatAnvil>` — the brand primitive

```vue
<FlatAnvil
  :color="'var(--sap-blue-7)'"
  :viewBox="[605, 297]"
  :rotate="0"
/>
```

Renders the canonical path `M0 0 0 297 307.459 297 605 0 0 0 Z` (605×297 viewBox). Brand-rule check: `:rotate` is *only* allowed at 0° — rotating the anvil violates SAP brand. The prop exists for future-proofing but throws a dev-mode warning if non-zero.

### 5.3 `<AnvilGrid>` — Pattern 2 (Anvil Grid Pattern)

Tiles `FlatAnvil` instances on a configurable grid (default 12-col × 4-row, matching POTX slide 14 specimen). Two-color: `:darkColor` (Blue 11) for the band background, `:lightColor` (Blue 7) for the anvils. Used as a header band on Team layout.

### 5.4 `<HarveyBall>` — the rating glyph

```vue
<HarveyBall :value="3" :of="8" :color="'var(--sap-blue-7)'" />
```

Rendered as a circle with a sector pie filled from 0° to `(value/of) × 360°`. Sourced from POTX slide-master shapes `Harvey 0/N` through `Harvey N/N` (where N ∈ {3,4,8}). Three sizes via `:size="'sm'|'md'|'lg'"` (16/24/32 px).

### 5.5 `<SapIcon>`

```vue
<SapIcon name="wrench" :color="'var(--sap-blue-7)'" />
```

Resolves to `theme/styles/_extracted/media/icons/{name}.svg` (POTX `image28.svg` = wrench, `image50.svg` = family-care, `image70.svg` = chart, etc.). The catalog is built at extraction time by `parse-potx-media.mjs`, which reads each SVG's first `<g id="...">` semantic name (the POTX preserves these — `id="wrench"`, `id="family-care"`, `id="full-stacked-column-chart"`).

The catalog is emitted as `theme/styles/_extracted/icons.json`:

```jsonc
{
  "wrench":                     { "src": "/sap/icons/wrench.svg",                     "viewBox": "0 0 43.18 43.18" },
  "family-care":                { "src": "/sap/icons/family-care.svg",                "viewBox": "0 0 96.21 96.21" },
  "full-stacked-column-chart":  { "src": "/sap/icons/full-stacked-column-chart.svg",  "viewBox": "0 0 96.21 96.21" }
  // … ~20 entries total, populated at extraction time …
}
```

`<SapIcon>` imports `icons.json`, resolves `name` to `src`, and renders an `<img>` with the configured viewBox (or inline SVG when `:color` overrides the default). Authors discover available names via `npm run icons:list` (prints the catalog) and via the CONTENT-GUIDE.md `<SapIcon>` section (auto-generated from `icons.json`). Unknown icon names render an empty placeholder + dev-mode warning.

### 5.6 `<ClassificationFooter>`

```vue
<ClassificationFooter :level="'INTERNAL'" :scope="'SAP and External Parties under NDA Only'" />
```

Renders a styled footer matching POTX: 8pt, grey-7, footer-position-12 (bottom-left, x=footer-x%, y=98%, derived from cover-tokens). Levels: `PUBLIC`, `EXTERNAL`, `INTERNAL` (default), `CONFIDENTIAL`. Shorthand: `:level="'PUBLIC'"` renders `"PUBLIC"` only (no scope). `:level="null"` hides the footer entirely.

### 5.7 `<Bio>` — refit to POTX slide-14 composition

v0.2 ships a `<Bio>` component. v0.3 redesigns it to match POTX slide 14 (`Meet our team`):

## 6. POTX Extraction Pipeline (v0.3)

`scripts/extract-brand.mjs` orchestrates four parsers and three emitters in a clear two-stage flow: parsers produce intermediate JSON in-memory; emitters consume the merged intermediate state to write final files (CSS, JSON manifests, copied media).

```text
SAP_Corp.potx
  │
  ├─ STAGE 1: parse (read POTX → intermediate JSON, no file writes) ───
  │
  │  ├── parse-potx-palette.mjs    → { colors: [{name, hex, textOn}…] }
  │  ├── parse-potx-layouts.mjs    → { layouts: { slideLayoutN: { name, type, … } } }
  │  ├── parse-potx-specimens.mjs  → { specimens: [{ slideN, layout, dynamicShapes, … }] }
  │  └── parse-potx-media.mjs      → { media: { ripple, flatAnvil, wordmark, photos, icons } }
  │
  ├─ STAGE 2: emit (intermediate JSON → final on-disk files) ──────────
  │
  │  ├── emit-brand-tokens.mjs       → theme/styles/_extracted/brand-tokens.css
  │  │                                 (named CSS variables + v0.2 deprecation aliases)
  │  ├── emit-layout-manifest.mjs    → theme/styles/_extracted/layouts.json
  │  │                                 (merges parse output with v0.3 status decisions
  │  │                                  from a hand-curated decisions.yaml — see below)
  │  ├── emit-icons-catalog.mjs      → theme/styles/_extracted/icons.json
  │  ├── emit-agenda-defaults.mjs    → theme/styles/_extracted/agenda-defaults.json
  │  └── emit-media.mjs              → theme/styles/_extracted/media/{ripple,anvil,…}
  │                                  → public/sap/* (build-time copy)
  │
  ├── emit-cover-tokens.mjs       ← UNCHANGED from v0.2 (geometry only)
  └── emit-typography-tokens.mjs  ← UNCHANGED from v0.2 (font sizes only)
```

**Parser/emitter boundary:** parsers are pure read-only POTX → JSON-in-memory; they never write files. Emitters take the merged parse output (plus `scripts/decisions.yaml`, a hand-curated file recording v0.3's ship/alias/exclude decisions per layout) and write the final artifacts. The split makes parsers easy to unit-test (give XML, get JSON) and lets a future `extract-brand.mjs --dry-run` print everything without touching disk.

`scripts/decisions.yaml` is the only non-generated file in this pipeline. It encodes the §4.5 manifest table (status per `slideLayoutN.xml`). When a future POTX adds a new layout, the build fails until a decision is added — preventing silent regressions.

Each parser/emitter is unit-tested with `vitest`. The full pipeline is integration-tested against `SAP_Corp.potx` (gitignored), with `describe.skipIf(!HAS_POTX)` for CI.

- POTX → tokens roundtrip: extract from `SAP_Corp.potx`, verify `brand-tokens.css` contains `--sap-blue-7: #0070F2`. Skipped in CI without POTX (`describe.skipIf(!HAS_POTX)`).
