# v0.4.0 Cover Rebuild + Polish — Design

**Status:** Draft for review
**Date:** 2026-06-14
**Author:** Thomas Jung (with Claude assistance)
**Supersedes:** none — extends v0.3.1
**Audit basis:** [docs/superpowers/audit/2026-06-14-v0.4-findings.md](../audit/2026-06-14-v0.4-findings.md) (committed on `audit/v0.4-layout-fidelity` branch as `audit/findings.md`)
**Scope split:** v0.4.0 ships P-1/A-1/2/3 (cover rebuild for variants a, k, l only) + P-2 (slide-overview close) + P-3 (Bio avatar). v0.4.1 ships remaining cover variants b–j plus P-4 (agenda) and P-5 (Bio team). v0.5+ ships feature gaps A-10 (DataPoint) and A-11 (NumberedCards).

## 1. Summary

v0.4.0 ships three things: (1) two trivial polish fixes (Bio avatar size, slide-overview close-button positioning); (2) a foundational rebuild of the Cover layout using the L/R-split architecture verified against POTX specimens, shipping variants `a`, `k`, `l` only; (3) a new `<PartnerLogoPlaceholder>` component giving authors a sanctioned way to render a partner-co-branding box on covers.

The cover rebuild is the substantial piece. v0.3's `<RipplePattern>` (a footer band) is conceptually wrong — POTX puts the Anvil Grid Pattern on a right-half column, not at the bottom. v0.4.0 deletes RipplePattern, ships a new `<AnvilGridDecoration>` that tiles via CSS `background-image: repeat`, and a new `<CoverDecoration>` that consumes a variant-config table to compose the L/R halves.

## 2. Goals

1. **Fix the cover defects** flagged in the audit (P-1, A-1, A-2, A-3) by establishing the correct L/R-split architecture matching POTX.
2. **Validate the architecture on 3 variants** (a, k, l) before scaling to 12 in v0.4.1.
3. **Introduce `<PartnerLogoPlaceholder>`** as a first-class component matching the POTX dashed-border placeholder, with three states: shown, placeholder, hidden.
4. **Trivial fixes** P-2 (slide-navigator close) and P-3 (Bio avatar) bundled in.
5. **Clean break from v0.3 RipplePattern** — delete the wrong abstraction rather than deprecate it (pre-v1, no production decks depend on it).

## 3. Non-goals

### 4.2 `CoverDecoration` config-table approach

`<CoverDecoration>` is a single component that takes the variant letter and an optional image, then internally consults `COVER_VARIANTS` to decide what to render. Rationale: 12 variants share the same L/R-split, the same content composition, and the same partner-logo box — they only vary in colors and whether the R-half has a photo. A single router with a config table scales cleanly to v0.4.1's 12 variants without component-file sprawl.

```ts
// theme/setup/cover-variants.ts

export type CoverLetter = 'a' | 'k' | 'l'  // expanded to 'a'-'l' in v0.4.1

export interface CoverVariantConfig {
  // L-half styling
  lBg: string                  // CSS color (e.g. '#ffffff' or 'var(--sap-blue-7)')
  textOnL: 'light' | 'dark'    // determines logo + text colors

  // R-half decoration
  rDecoration: 'anvil-grid' | 'flat-anvil-single'
  rBg: string                  // CSS color OR 'photo' (photo fills R)
  hasPhoto: boolean            // requires `image:` front-matter
  anvilColor: string           // CSS color for the anvil shapes
}

export const COVER_VARIANTS: Record<CoverLetter, CoverVariantConfig> = {
  a: {
    lBg: '#ffffff', textOnL: 'dark',
    rDecoration: 'anvil-grid', rBg: 'var(--sap-blue-11)',
    hasPhoto: false, anvilColor: 'var(--sap-blue-6)'
  },
  k: {
    lBg: 'var(--sap-blue-7)', textOnL: 'light',
    rDecoration: 'anvil-grid', rBg: 'photo',
    hasPhoto: true, anvilColor: 'rgba(255,255,255,0.4)'
  },
  l: {
    lBg: '#ffffff', textOnL: 'dark',
    rDecoration: 'flat-anvil-single', rBg: 'var(--sap-blue-2)',
    hasPhoto: true, anvilColor: 'var(--sap-blue-7)'
  }
}

export function getVariantSpec(v: string): CoverVariantConfig
export function resolveCoverVariant(input?: string): CoverLetter
```

### 4.3 L/R proportion

Layout uses **60/40** — content (L) gets 60% width, decoration (R) gets 40%. Validated against POTX cover-a and cover-k renders. v0.4.1 may revisit per-variant if cover-L specifically reads wrong.

### 4.4 `AnvilGridDecoration` mechanics

- Generates a small SVG tile at build time as a data-URL OR ships a static SVG file in `public/sap/anvil-tile.svg`. The tile contains one Flat Anvil shape (the canonical 605×297 path) inside a square viewBox.
- Applied as `background-image: url(...); background-repeat: repeat; background-size: 60px 30px` on a positioned div sized to the R-half (40% slide width × 100% slide height).
- Color override via CSS variable `--anvil-color` consumed by the SVG's `fill` attribute. Set per-variant from `COVER_VARIANTS[v].anvilColor`.
- Tile rendered at 2× logical pixel size (e.g., 120×60 viewBox for a 60×30 visual tile) so retina displays render the anvils crisply rather than upscaling.
- No DOM cost beyond the wrapper div.

### 4.5 `PhotoFrame` (kept from v0.3)

Reused as-is for variants k and l. The 40%-wide R-half column passes width via CSS — `<PhotoFrame>` already adapts via `object-fit: cover`. If real cover-k/l rendering reveals a fit issue, scope a small fix into the implementation task; do not refactor PhotoFrame in v0.4.0.

### 4.6 `PartnerLogoPlaceholder` — three-state component

```vue
<!-- Usage -->
<PartnerLogoPlaceholder :logo="frontmatter.partnerLogo" :textOnL="spec.textOnL" />
```

**Render branches:**

| Prop value | Renders |
| --- | --- |
| `undefined` (omitted from front-matter) | Dashed-border box, ~12% slide width × 8% slide height. Border color follows `:textOnL` (`light` → white dashed; `dark` → grey-7 dashed). Text "Add partner logo and alt text" centered, matching POTX wording verbatim. |
| `<string>` (URL) | `<img :src="logo" alt="Partner logo">` sized to fit the same box. |
| `null` | `v-if="logo !== null"` → renders nothing. |

## 5. Author-facing API

```yaml
---
layout: cover
variant: a              # one of: 'a', 'k', 'l' (v0.4.0); 'b'-'j' arrive in v0.4.1
title: Building Cloud-Native Apps with SAP CAP
presenter: thomas-jung  # resolves to Name from presenters/<slug>.yaml
event: SAP TechEd 2026  # optional; falls back to event.yaml
date: 2026-09-15        # optional; falls back to today

# Variant k and l require this:
image: /covers/keynote-photo.png

# Optional partner-logo control (three states):
partnerLogo: /covers/partner-acme.png   # state 1: shown
# (omitted)                             # state 2: dashed placeholder
partnerLogo: null                       # state 3: hidden entirely

classification: PUBLIC  # existing v0.3 footer control
---
```

| Field | Type | Required | v0.3 → v0.4 change |
| --- | --- | --- | --- |
| `layout` | string | yes | unchanged: still `'cover'` |
| `variant` | `'a' \| 'k' \| 'l'` | yes | v0.3 had `'a'..'l'` (12); v0.4.0 ships only 3, others throw build-time error |
| `title` | string | yes | unchanged |
| `presenter` | string (slug) | optional | unchanged; resolves to Name + Title |
| `event` | string | optional | unchanged; falls back to `event.yaml#name` |
| `date` | string (ISO 8601 date) | optional | NEW. Renders below presenter Name. Falls back to today's date in deck timezone. POTX shows "Month Date, Year" format — we follow via `Intl.DateTimeFormat('en-US', { dateStyle: 'long' })`. |
| `image` | string (path) | required for `k`/`l`, ignored for `a` | semantics tighten: previously optional everywhere |
| `partnerLogo` | string \| null | optional (default: placeholder) | NEW. Three-state per §4.6. |
| `classification` | string | optional | unchanged from v0.3 |

### 5.1 Build-time validation

- `variant: 'b'` (or any unimplemented variant) → throws at build with: `"Cover variant 'b' is not yet implemented in v0.4.0. Available: a, k, l. Tracked for v0.4.1 in audit/findings.md."`
- `variant: 'k'` or `'l'` without `image:` → throws: `"Cover variant '<x>' requires an image: front-matter. Add image: /path/to/photo.png (relative to public/)."`
- `variant: 'a'` with `image:` set → silent ignore (don't fail; warn in dev console only).

### 5.2 Migration path from v0.3 to v0.4.0

- Decks using `variant: a` continue to work (text content + logo unchanged; visual changes from footer-Ripple to right-half AnvilGrid).
- Decks using `variant: b`-`j` will break the build — need to be migrated to `a` (or wait for v0.4.1).
- Decks using v0.3 `variant: a` with `image:` set: image is ignored, doesn't break the build.
- v0.3 `presenter` + `event` keys still work; new `date` key is additive.

### 5.3 Documentation

[CONTENT-GUIDE.md](../../CONTENT-GUIDE.md) gets a new "Cover" section describing the L/R composition, the three variants, the partner-logo control, and the 14-word soft cap on titles (see §6.2 risks).

### 6.1 Visual regression strategy

**Per-task rebaseline, not full sweep.** Unlike v0.3 Task 34 (which had to rebase 99 slides because 30+ tasks landed), v0.4.0 changes a small known set of slides. The plan should:

1. Run `npm run test:visual` after each task that changes a visible layout.
2. Only rebase the snapshots that genuinely diverge for that task's reason.
3. Manual visual sanity check on cover-a, k, l before approving the rebase.

The 16 clear-space tests in `tests/visual/clear-space.spec.ts` remain excluded (deferred to v0.4 navigation work — out of v0.4.0 scope).

## 7. Risks

| Risk | Impact | Mitigation |
| --- | --- | --- |
| The 60/40 L/R split at 1280×720 cuts off long titles | Med | Title region is 60% × ~25% of slide = ~770×180 px. Two lines of 72-Bold at ~80 px line-height fit ~5–7 words/line. Document a 14-word soft cap in CONTENT-GUIDE; truncate cleanly with CSS `line-clamp` on overflow. |
| `background-image: repeat` of a Flat Anvil tile renders pixelated on retina displays | Med | Generate the tile SVG at 2× logical size (e.g., 120×60 viewBox for a 60×30 visual tile). Browsers downscale crisply; upscale would pixelate. |
| Removing v0.3's `<RipplePattern>` breaks any deck that imported it directly (third-party usage) | Low | Pre-v1; no production decks rely on theme internals. Document the removal in CHANGELOG migration notes. The `cover-variants.ts` consumer is updated as part of this task. |
| `partnerLogo: null` YAML edge case (Slidev/JS-YAML inconsistency) | Low | YAML's `null` and JS's `null` align; tested in YAML data parsing already. |
| New `date:` field default-to-today produces inconsistent rendered dates across re-publish builds | Low | Document that a missing `date:` falls back to "build-time current date." Authors who want stability should set it explicitly. Render with `Intl.DateTimeFormat` — locale-stable. |
| Per-variant CSS lives in `cover.vue`'s `<style scoped>` — at v0.4.1 (12 variants) the file gets long | Med | Pre-empt by extracting `cover-variants.css` (a generated file from `COVER_VARIANTS`) so v0.4.1's 9 additional variants append, not edit, the CSS. v0.4.0 ships the generator from day one. |
| `<PhotoFrame>` (kept from v0.3) doesn't currently support being framed inside a 40%-wide column | Low | Test cover-k and cover-l manually after wiring; if the existing component needs adjustment, scope a small PhotoFrame.vue tweak into the cover-k/l implementation task rather than a v0.4.1 deferral. |

## 8. Open questions

All deferred to implementation phase, not blocking spec approval:

1. **Tile size.** §4.4 has `background-size: 60×30px` as a placeholder. The actual visual density depends on what reads as "POTX-correct" — judgment call after a render comparison once the component is built.
2. **Anvil color in cover-K.** Tentatively `rgba(255,255,255,0.4)` (white at 40% opacity for the anvils overlaying a photo). May need adjustment based on photo contrast.
3. **Date locale.** `en-US` default; future i18n work in v0.5+.
