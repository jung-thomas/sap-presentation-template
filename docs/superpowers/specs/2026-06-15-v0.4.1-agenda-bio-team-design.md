### 4.2 Architecture

**Single layout file** (`theme/layouts/agenda.vue`) composing:

- **Left content region** — 67% width: `<AgendaTitle>` + `<AgendaList>` (one row per item: number + label + description, with hairline below each except the last).
- **Right decoration column** — 33% width: `<AnvilGridDecoration bg="var(--sap-blue-10)" color="var(--sap-blue-6)" />` (the v0.4.0 primitive, just placed in a different region).

No new components needed beyond a tiny `<AgendaItem>` if it makes the layout file cleaner. Reuse `<ClassificationFooter>` as in v0.4.0.

### 4.3 Author API

**Unchanged from v0.3:**

```yaml
---
layout: agenda
title: Agenda
items:
  - title: Foundations of CAP
    description: Optional description text  # NEW: rendered below title
  - title: The data modelling layer
    description: Optional description text
  - ...
classification: PUBLIC
---
```

**Additions:**

- `description` field on each item is now first-class (matches POTX's "Add a short description of Agenda item..." placeholder text). If omitted, the description row is suppressed.
- Existing v0.3 `agenda` config object (`toc`, `showSubsections`, `dividers`, `sectionNumbers`, `slideNumbers`) is preserved for backward compatibility, but `sectionNumbers` is hardcoded `true` in v0.4.1 (POTX always shows them) — the field is read but its value is ignored.

**Backward compatibility — `subsections` vs `description`:**

v0.3 supported `items[].subsections: string[]` with the `agenda.showSubsections: true` flag to render bulleted child rows. v0.4.1 keeps that path working unchanged for existing decks. The new `description` field is additive:

- Item with only `description` → renders POTX-style description row (single grey line of text)
- Item with only `subsections` (and `showSubsections: true`) → renders v0.3-style nested bullet list
- Item with both → renders the `description` row first, then the `subsections` bullet list. Authors who use both are consciously stacking them; we don't error.
- Item with neither → label-only row (POTX-valid; description placeholder text not synthesized)

The `subsections` rendering still uses muted grey (no POTX target to match — it's our extension), while the new `description` matches POTX black. The two visually coexist and don't fight.

**Removed:** the `variant: a` / `variant: b` distinction. POTX agenda-a is just a blank slide; the new layout always renders the agenda-b composition. The `agenda-layout--b` CSS class is dropped.

### 5.2 Architecture

**Single component update** (`theme/components/Bio.vue`'s team-mode branch). The team-mode template composes:

- `<AnvilGridDecoration bg="var(--sap-blue-10)" color="var(--sap-blue-6)" class="bio-band" />` constrained to top 50% via scoped CSS (`.bio-band { position: absolute; top: 0; left: 0; right: 0; height: 50%; }`).
- `<div class="bio-cards">` containing exactly 4 `<article class="bio-card">` elements via a fixed `grid-template-columns: repeat(4, 1fr)` with 47px (2.4%) gap and 80px (4.2%) outer margins. The grid is positioned at `top: 24.4%; bottom: 12%;` so cards naturally overlap the band by 25.6%.

Each card renders: photo (top, 4:5 portrait — implemented as `<img>` inside an `aspect-ratio: 4/5` wrapper with `object-fit: cover` so wide source photos crop to portrait) → name (72-Bold black) → role (grey) → QR code (fixed 64×64 bottom-right of card text region).

Replaces v0.3's `<AnvilGrid>` (the multi-cell grid component with `position: absolute; inset: 0`) and the `auto-fit, minmax(180px, 1fr)` card grid.

**Title rendering decision:** the `<Bio>` component does NOT render a title of its own. The host slide is responsible for the "Meet our team" heading. The conventional way to wire this is `layout: title-only` with `title: 'Meet our team'` in the front-matter, then `<Bio :people=...>` as the slide body. The team-mode CSS positions the Bio decoration band so the host's `<h1>` (provided by the title-only layout) lands inside the band naturally.

For v0.4.1 to actually match POTX slide 14's white-on-band title, the `title-only` layout's `<h1>` must be styled to render white when followed by a Bio team-mode child. This is handled via a CSS rule on `.title-only:has(.bio--team) h1 { color: #fff; }` (modern `:has()` selector, supported in all current Chromium / WebKit / Firefox). No prop API change to `<Bio>`; no changes to `title-only.vue` template; just a CSS rule keyed off the team-mode class.

### 5.3 Author API

**Unchanged from v0.3:**

```html
<Bio :people="[
  { name: 'Nina Thompson', role: 'Data Analyst', photo: '/team/nina.jpg', qr: 'https://...' },
  { name: 'Marcus Bennett', role: 'CSM', photo: '/team/marcus.jpg', qr: 'https://...' },
  { name: 'Sofia Nguyen', role: 'QA Engineer', photo: '/team/sofia.jpg', qr: 'https://...' },
  { name: 'Ethan Brooks', role: 'Tech Sales', photo: '/team/ethan.jpg', qr: 'https://...' }
]" />
```

**Constraint:** exactly 4 people. With ≠ 4, the layout still renders but visually breaks the POTX intent. v0.4.1 emits a `console.warn` during dev when `people.length !== 4` (no build error — authors of 3-person teams need it to render rather than throw).

**Title:** the `<Bio>` component does NOT render its own title. Host the team slide via `layout: title-only` (POTX convention) with the heading in front-matter. See §5.2 for the CSS rule that styles the `title-only` `<h1>` white when a Bio team-mode child is present.

## 6. AnvilGridDecoration tile period

POTX's anvil density on agenda-b's right column and slide 14's header band looks visually similar to v0.4.0's existing 60×30px tile, but a precise side-by-side has not been done. Rather than punt this as an open question, **v0.4.1 commits to no API change in this release**:

- v0.4.0's `<AnvilGridDecoration>` keeps its current props (`color`, `bg`) and hardcoded `background-size: 60px 30px`.
- The first task of Stage A is a manual side-by-side render comparison (using the same Slidev dev server we used for POTX measurement). If the v0.4.0 tile reads as visually wrong density-wise vs POTX, the implementer either:
  - Patches `<AnvilGridDecoration>`'s scoped CSS to use a different fixed `background-size` for v0.4.1 (acceptable: the cover already passes density inspection in v0.4.0, so a small tweak that doesn't break the cover is fine), or
  - Surfaces the issue back to the spec author for a follow-up release with a new `tileSize` prop (NOT done in v0.4.1).
- No `tileSize` prop is added in v0.4.1. The component's API surface stays as-is.

## 7. Hotfix carry-overs

### 7.1 AnvilGridDecoration restore (PR #8)

The empty-file regression on main caused by commit `d177159` is fixed by `hotfix/v0.4.0.1` PR #8. **Merge order is strict:**

1. PR #8 merges to `main` first (and tags v0.4.0.1 if a tag is desired).
2. The `spec/v0.4.1` branch then rebases on the new `main` HEAD. The `fc5ced7` commit on the v0.4.1 branch becomes a no-op against the rebase target (the file is already restored on main from PR #8) — git detects this automatically and drops the duplicate commit.
3. v0.4.1's PR opens against the post-PR#8 main.

This avoids a merge conflict on `AnvilGridDecoration.vue` and keeps the v0.4.1 PR diff focused on agenda + bio team work without re-shipping the same restoration.

### 7.2 P-2 close button (commit `d0df851`)

The Slidev "Goto..." dialog fix is on the v0.4.1 branch. Two parts:

1. CSS in `theme/styles/index.css` constrains the dialog max-height and the autocomplete list max-height so they stay inside the viewport.
2. JS in `theme/setup/main.ts` injects a real `<button class="theme-goto-close">×</button>` that calls Slidev's exported `showGotoDialog` ref to close the panel cleanly.
