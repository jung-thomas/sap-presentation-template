## Summary

The v0.4.4 rebaseline produced **8 modified + 45 new = 53 changed baselines** out of 146 total. The remaining 94 baselines are byte-identical to v0.3.

The 94 unchanged baselines are NOT a v0.4.4 regression. They reflect **pre-existing source-markdown bugs** in `pages/all-layouts.md`: the affected slides have front-matter delimiter patterns that Slidev's parser does not recognize, so it falls back to its built-in default layout and renders the raw front-matter as visible body text. The v0.4.4 capture matches the v0.3 capture byte-for-byte because both flow through the same Slidev default-layout fallback.

The rebaseline IS valid: it captures the actual current rendering of every slide. v0.4.4's stated scope (spec section 5 non-goals) explicitly defers POTX-fidelity repairs to v0.4.5 / v0.5. The 94 broken-front-matter slides are this release's natural follow-up.

Layout-map evidence (Step 1 of Task 5): only 3 of 146 slides resolve to a `theme/layouts/<name>.vue` import (slides 1 = cover, 73 = text-with-icons, 74 = table). All other 143 slides resolve to `@slidev/client/layouts/default.vue`. Of those 143, a smaller subset compose at least one `theme/components/<Name>.vue` (Bio, CodeBlock, Agenda, DemoCallout, QRCode, Roadmap, EventBadge, Speaker), and those produce actual themed output even though the layout itself is the Slidev default. The remainder render only Slidev's default-layout fallback over the raw markdown body.

## Source-debt findings (deferred to v0.4.5 / v0.5)

The 94 unchanged baselines were not all spot-checked individually. Instead, they form a class of issue documented here for follow-up.

### Pre-existing front-matter parsing bugs across `pages/all-layouts.md`

- **What:** for ~94 of the 146 gallery slides, the front-matter delimiter pattern in the markdown source is not being recognized by Slidev's parser, so the slide compiles to use Slidev's built-in default layout instead of the declared `layout:`. The rendered baseline shows raw `layout: foo title: "..."` as visible body text instead of the SAP-themed layout.
- **Concrete example A:** slide URL 4 (source declares `layout: cover` variant `b`). The PNG literally renders `layout: cover title: "Cover B" subtitle: "Subtitle text" variant: b presenter: thomas-jung` as page content. Compiled module imports `@slidev/client/layouts/default.vue`.
- **Concrete example B:** slide URL 50 (source declares `layout: content-image-4col`). Compiled module imports `@slidev/client/layouts/default.vue` instead of `theme/layouts/content-image-4col.vue`. Verified via `curl http://localhost:3031/all-layouts.md__slidev_50.md?import` during Task 5 Step 1.
- **Concrete example C:** slides 56, 58, 60, 62 (source declares `full-bleed-image`, `image-slide`, `title-image-third`, `title-photo` respectively). All compile to default layout; baselines show only the placeholder image with literal layout name as title text.
- **Source pattern likely at fault:** `---` (slide separator) followed immediately by `layout: foo` without a second `---` to close the front-matter, then body. This pattern parses inconsistently in Slidev's markdown plugin. Fixing requires auditing each slide's front-matter delimiters — explicitly deferred to v0.4.5 / v0.5 per v0.4.4 spec section 5 non-goals.
- **How to identify in v0.4.5:** any `slide-NN.png` baseline byte-identical to its v0.3 counterpart, AND whose compiled module imports `@slidev/client/layouts/default.vue` rather than `theme/layouts/<name>.vue`. The regression smoke test (`tests/regression/theme-path.spec.ts`) only pins slides 74 and 82; expanding it to assert layout imports for representative slides of every category is itself a v0.4.5 candidate.
- **Diagnostic command (re-runnable):** with the gallery up at port 3031, `curl -s "http://localhost:3031/all-layouts.md__slidev_<N>.md?import" | grep -oE '(theme[\\/]layouts[\\/][a-z0-9-]+|@slidev[\\/]client[\\/]layouts[\\/]default)\.vue'` for each slide N. Anything matching the second branch is a candidate.
